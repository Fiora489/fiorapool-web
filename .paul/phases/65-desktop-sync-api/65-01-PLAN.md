<objective>
Ship the web-side API surface the desktop client will read/write against, with per-user API keys, documented JSON contracts, and a small pairing page where the user generates + revokes keys. No changes to the main Creator or Hub UIs.
</objective>

<scope>
**IN:**
- Migration: `desktop_api_keys(id, user_id, name, key_hash, last_used_at, created_at, revoked_at)` with RLS owner-only + bcrypt-hashed key storage
- `api-keys.ts` — `generateDesktopApiKey(userId, name)` returning `{ id, plaintextKey }` (plaintext returned once); `listKeys(userId)`; `revokeKey(id)`
- `auth.ts` — `authenticateDesktopRequest(req): { userId, keyId } | null` reading `Authorization: Bearer <key>`
- Routes (Node runtime, not edge — need service role for cross-user reads where applicable):
  - `GET /api/desktop/builds` — list current user's builds (summary list)
  - `GET /api/desktop/builds/[id]` — full build with all joined children (items, runes, matchup notes, item swaps, skill order, combos) — returned as the canonical "export" JSON shape the desktop will translate to League's itemset.json
  - `POST /api/desktop/match-tags` — desktop pushes detected matches from LCU integration:
      body: `{ matchId, buildId, won }`
      inserts into `build_match_tags` (idempotent), re-runs WR rollup
  - `GET /api/desktop/meta` — returns current patch, item catalogue version, summary of user's XP/level for the desktop status bar
- Rate limit 60 req/min per API key
- Errors use RFC 7807 problem+json
- Small pairing page `/builds/desktop-sync`:
  - List of active keys with name, created_at, last_used_at
  - "Generate new key" button → reveals plaintext once + copy-to-clipboard
  - "Revoke" button per key
- Docs block on the pairing page showing endpoint list + example curl commands (so the desktop dev can integrate)

**OUT:**
- The desktop client itself
- Champ-select autodetect (needs LCU + spectator-v5 — future milestone)
</scope>
