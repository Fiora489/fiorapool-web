<objective>
Post-match auto-tagging of builds + WR rollup. When the user plays a match whose items path overlaps ≥ 80 % with one of their saved builds (same champion), that match is linked to the build via `build_match_tags`. Personal WR and aggregate (opt-in) WR are computed from these tags.
</objective>

<scope>
**IN:**
- `match-tagger.ts`:
  - `computeItemOverlap(matchItemsFinal[], buildItemsFinal[]): number` — Jaccard on final-build item sets, items only (not order)
  - `detectMatchBuildTags(userId, match)` — returns `{ buildId, overlap }[]` for all candidate builds
  - `applyMatchBuildTags(userId, matchId, tags)` — inserts into `build_match_tags`; idempotent
  - Threshold constant exported: `BUILD_MATCH_THRESHOLD = 0.8`
- `build-wr.ts`:
  - `getPersonalBuildWr(buildId, userId): { games, wins, winrate }`
  - `getAggregateBuildWr(buildId): { games, wins, winrate } | null` (null if < 5 tags or opt_in_aggregate = false)
  - `getBuildPerformanceHistory(buildId, userId, { limit })` — last N tagged matches for the chart in UI phase
- Migration:
  - Add view `build_wr_personal(build_id, user_id, games, wins, winrate)`
  - Add view `build_wr_aggregate(build_id, games, wins, winrate)` — only rolls up rows from builds with `opt_in_aggregate = true`
  - Indexes listed in Phase 53 already cover this; add `(user_id, detected_at desc)` on `build_match_tags` for history
- Hook into existing match sync pipeline (`src/lib/matches/sync.ts` — or wherever match-v5 ingest lives): after inserting a new match row for the user, call `detectMatchBuildTags` → `applyMatchBuildTags`. Fire-and-forget; wrap in try/catch so build tagging never breaks match sync.

**OUT (→ UI phases):**
- Build WR badge on build card (Phase 66 or 67 — depends on viewer)
- Personal performance history chart
- Aggregate WR display on public build pages
- Opt-in toggle UI (the flag itself flips via existing `setBuildPublic`/`opt_in_aggregate` action from Phase 53)
</scope>
