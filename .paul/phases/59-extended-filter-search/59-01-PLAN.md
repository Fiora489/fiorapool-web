<objective>
Extend the Phase 58 hub query layer with the advanced search/filter contract the Build Hub UI (Phase 65) will consume: typo-tolerant search, boolean operators, search analytics for trending, URL-state polish, and aggregate facet pre-compute for the filter sidebar. No new UI.
</objective>

<scope>
**IN:**
- `search-advanced.ts` — layered on top of `hub-query.ts`:
  - `advancedQuery({ q, filters, sort, mode })` where `mode: 'all-words' | 'any-word' | 'exact-phrase'`
  - Trigram typo tolerance — fallback to `similarity(name, q) > 0.3` when tsvector returns 0 rows
  - Boolean operators in raw query: `+word`, `-word`, `"exact phrase"`, parsed into `tsquery`
  - Highlighted snippets via `ts_headline` (returned per row for the UI to render)
- `trending.ts` — `getTrendingSearches(window='7d', limit=10)`. Reads from `search_analytics` table aggregated by query + period. `getTrendingBuilds(window='7d')` ranks by Δ bookmarks + Δ views.
- `search-analytics.ts` + migration creating `search_analytics(id, user_id, query_hash, query_text, filters_json, ts)` — logged on every search (respects user opt-out flag from settings). RLS: user can read own rows; aggregates exposed via view `trending_searches_view`.
- Facet pre-compute: add stored function `hub_facet_counts(query jsonb) returns jsonb` returning champion/role/patch/tag/keystone counts for a query subset — avoids N facet queries from the UI
- URL-state enhancements in `hub-query.ts`:
  - `parseHubQuery` tolerates partial / legacy params; never throws
  - `serializeHubQuery` omits defaults
  - Canonical ordering of filter params for cache-friendly URLs
- Similar-builds endpoint refinement (builds on Phase 58 `similarity.ts`): `listSimilarPublic(buildId, limit)` respects rank-floor + freshness filters of the current viewer

**OUT (→ Phase 65 UI):**
- `<SearchBar>`, `<FilterSidebar>`, `<SortDropdown>`, `<SavedSearchesDrawer>`, `<SimilarBuildsRail>`, `<TrendingWidget>`, autocomplete popover — all in the UI phase
</scope>
