<objective>
## Goal
Turn `/analytics/clutch` into a real clutch-performance dashboard: clutch-rate overview, clutch-type breakdown (comeback / long-game / lane-loss-recovery / stomp), behind-at-10 win rate (the truest clutch signal), per-champion clutch wins, and a rich examples list.

## Purpose
The current page only counts "wins where game >28min OR negative gold@10" — a fuzzy combined metric. Players want to know: "When I'm behind in lane, do I actually come back?" This phase answers that with a behind-at-10 conditional win rate plus a typed breakdown of clutch wins.

## Output
- Enhanced `computeClutch()` with clutch-type categories, behind-at-10 analysis, per-champion stats, richer examples
- 5 new components
- Rewritten page with 5 sections
</objective>

<acceptance_criteria>

## AC-1: Clutch Type Breakdown
```gherkin
Given a user with ≥1 win
When computeClutch() runs
Then clutchTypes returns counts for { comeback, longGame, laneLossRecovery, stomp }
     where:
     - comeback     = win AND gold_diff_at_10 ≤ -500
     - longGame     = win AND game_duration_seconds > 28*60
     - laneLossRecovery = win AND cs_diff_at_10 ≤ -10
     - stomp        = win AND gold_diff_at_10 ≥ +1500 AND game_duration_seconds < 25*60
     A single win can match multiple types.
```

## AC-2: Behind-at-10 Conditional Win Rate
```gherkin
Given matches with gold_diff_at_10 populated
When computeClutch() runs
Then behindAt10 returns { games, wins, winRate } for matches where gold_diff_at_10 < -500
And aheadAt10 returns the same for matches where gold_diff_at_10 > +500
```

## AC-3: Per-Champion Clutch Wins
```gherkin
Given multiple clutch wins on different champions
When computeClutch() runs
Then clutchChampions returns array of { champion, clutchWins, types: string[] }
     sorted by clutchWins desc, max 10
```

## AC-4: Examples Enriched
```gherkin
Given clutch wins
When the page renders
Then each example shows champion, duration, gold deficit/surplus,
     CS diff at 10, and the clutch type tag(s)
```

## AC-5: Mobile + Empty State
```gherkin
Given mobile viewport at 390px or 0 wins
When the page loads
Then layout has no horizontal overflow; empty state shows centered message when 0 wins
```
