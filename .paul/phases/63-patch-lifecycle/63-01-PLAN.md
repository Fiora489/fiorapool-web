<objective>
Data layer for patch-lifecycle features: staleness calculation, Data Dragon patch-to-patch item diff, and the bulk patch-bump query that powers the Creator UI's "refresh all my builds" workflow. No UI — Phase 66 renders against this.
</objective>

<scope>
**IN:**
- Migration: table `patch_snapshots(patch text pk, items jsonb, runes jsonb, spells jsonb, taken_at timestamptz)` — snapshot of Data Dragon catalogues per patch
- Worker `refreshPatchSnapshots()` — compares current Data Dragon version with the latest snapshot; if new, fetches + stores. Invoked via scheduled-task cron (daily) or manual
- `patch-diff.ts`:
  - `diffPatchItems(fromPatch, toPatch): ItemPatchDiff` — returns `{ stats_changed: id[], icon_changed: id[], removed: id[], added: id[] }` computed from snapshots
  - `diffPatchRunes(fromPatch, toPatch): RunePatchDiff`
- `staleness.ts`:
  - `isBuildStale(build, currentPatch): { stale: boolean, reason?: 'items-changed' | 'patches-old' | 'items-removed', patchesBehind: number }`
  - `STALENESS_PATCH_THRESHOLD = 2`
- `patch-bump.ts`:
  - `listBuildsNeedingBump(userId, currentPatch): BumpCandidate[]` — builds with items changed or ≥ 2 patches old
  - `BumpCandidate` includes `affectedItems: {buildBlockType, itemIds[]}[]` so UI can highlight
  - Action: `ackBuildPatch(buildId, newPatch)` — sets `last_validated_patch` and optionally `patch_tag` to current
- Reuses existing `stampLastValidated` from Phase 56

**OUT (→ Phase 66 UI):**
- `<StalenessBanner>` component on build cards
- `/builds/patch-bump` page with bulk actions
- Diff highlight UI on items
</scope>
